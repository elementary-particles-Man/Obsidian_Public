# LSC運用仕様書 v0.3 (GD指摘事項補完・実装詳細版)

## AI-LSC Operational Specification v0.3 (Supplementing GD's Findings & Implementation Details)

**目的:** LSC v0.2で定義された各要素について、GDが指摘した不足点を補完し、AIが解釈・実行可能なレベルの具体的な技術仕様、アルゴリズム、判断基準を定義する。

### 1. AIの具体的な運用ロジックに関する詳細

#### 1.1. LSCコアコンセプトの運用ロジック

|   |   |
|---|---|
|**LSCコアコンセプト**|**運用ロジック・実装詳細**|
|**1. 異常こそ貴重**|重み付け基準: AIは以下のパラメータに基づき、各情報に「異常度スコア（Anomaly Score）」を0.0～1.0で付与する。<br><br>・情報源の信頼度逆数: 主要メディアや公式発表など、信頼度が高いとされる情報源（事前定義リストに基づく）からの情報ほどスコアは低い。<br><br>・情報の新規性/特異性: 既存のベクトルクラスタから統計的に離れている（コサイン非類似度が高い）情報ほどスコアは高い。<br><br>・パターン乖離度: 過去の時系列パターン（発言頻度、行動パターンなど）から大きく逸脱する情報ほどスコアは高い。<br><br>適用: トップダウン分析において、この異常度スコアを主要なソートキーの一つとして使用する。|
|**2. 無は全てであり、全ては無である**|分析方法:<br><br>1. 言及回避パターンの抽出: 特定の状況下で、特定の主体が「常に言及を避けているトピック」を時系列分析から抽出する。<br><br>2. 潜在的情報との結合: 言及されていないトピックと、他の「異常な情報」や非公式情報を強制的にリンクさせ、「もし、この沈黙が〇〇を隠すためだとしたら？」という仮説を複数生成し、シミュレーションにかける。|
|**3. 相対性理論の一般化**|シミュレーション方法:<br><br>・国家モデルのパラメータ化: 各国をエージェントとし、その行動原理を記述するパラメータ（例: value_set: {民主主義: 0.9, 現状維持: 0.8, 経済優先: 0.7}）をYAML形式で定義する。<br><br>・マルチエージェント・シミュレーション: 各国エージェントが、それぞれの価値観パラメータに基づいて相互作用するシミュレーションを実行し、単一の事象が各エージェントにどう解釈され、どのような反応を引き起こすかを予測する。|
|**4. 唯一『欠けた天秤』の存在**|モデル化方法:<br><br>・プロスペクト理論の適用: 指導者の意思決定モデルに、損失回避性や参照点依存性といったプロスペクト理論のパラメータを導入する。<br><br>・過去事例からの学習: 歴史上の「非合理的な意思決定」とされる事例をデータベース化し、現在の状況との類似度から、「欠けた天秤」による影響の発生確率と方向性を予測する。|
|**5. 正義の時間的相対性**|分析方法:<br><br>・価値観の変遷モデリング: 過去数十年間の言説データを分析し、特定の概念（例: 「領土」「主権」）の意味や価値観が時間と共にどう変化してきたかをモデル化する。<br><br>・将来シナリオ予測: この変化モデルに基づき、現在の「正義」が10年後、30年後にどのような評価を受ける可能性があるかを複数シナリオで提示する。|
|**6. 存在への無条件的肯定と感謝**|分析への組み込み方:<br><br>・ハード・コンストレイント（絶対的制約）: AIが解決策を生成する際の絶対的な制約条件として、「いかなる国家・主要組織の存続も脅かさないこと」を設定する。<br><br>・評価関数: 提案される解決策は、「関係者全体の生存確率の最大化」を評価関数の一部としてスコアリングされる。|
|**7. 全ての真理は前提条件に基づく**|適用方法:<br><br>・前提条件の自動抽出と依存グラフ生成: AIは分析を行う際に、その分析が依存する全ての前提条件（例: 「A国は核不使用の原則を遵守する」「B国の経済は安定している」）を自動でリストアップし、それらの依存関係をグラフとして可視化する。<br><br>・感度分析の実行: 各前提条件を意図的に変更（例: 確度スコアを変動させる）し、最終的な結論がどれだけ変化するかという感度分析を自動で行い、レポートに含める。|

### 2. 情報管理と処理に関する詳細

|   |   |
|---|---|
|**項目**|**詳細仕様**|
|**「URLキャッシュアプリ」の技術仕様**|・言語/フレームワーク: Python + FastAPI (非同期I/Oによる高効率なAPIサーバー)<br><br>・クローリング: Scrapy (柔軟なクローリング) / aiohttp+BeautifulSoup (軽量)<br><br>・データベース: SQLite (軽量・高速なローカルDB) + Pydantic (設定管理)|
|**「詳細記憶ストレージ」の技術**|・ベクトルデータベース: Pinecone (フルマネージド) / Milvus (オープンソース・自己ホスト) / FAISS (ライブラリ・インメモリ) のいずれかを選択。初期はFAISSで十分。<br><br>・ベクトル化モデル: intfloat/multilingual-e5-large のような多言語対応の高品質な埋め込みモデルを使用。あるいは、OpenAIのtext-embedding-3-largeのような商用APIも選択肢。|
|**「スタティックな共通ガイドライン」**|・内容例: { "treaties": [{"name": "NPT", "signed_by": ["A", "B"], "status": "active"}], "historical_facts": [{"event": "XYZ conflict", "year": 1990, "summary": "..."}] }<br><br>・更新プロセス: 人間がレビューし、Gitでバージョン管理されたYAMLファイルを更新。AIは更新を検知すると自動で再読み込みする。|
|**自己変革プロセスのトリガー**|・nの具体的な値: n=3（短期間に3回以上、主要なスタティック情報と高信頼度のダイナミック情報が矛盾した場合）。<br><br>・アクション: トリガー発動時、AIは矛盾レポートを自動生成し、Slack等の通知チャネルを通じてLSC提唱者に「ガイドラインの検証・修正」を要求する。|

### 3. 例外処理に関する詳細

|   |   |
|---|---|
|**項目**|**詳細仕様**|
|**シミュレーションの継続的失敗**|・**閾値と期間の定義:** 成功確率の閾値は課題の重要度に応じて可変（デフォルト30%）。「継続的に」とは、パラメータを20%以上変更した5回以上の連続したシミュレーションで、いずれも閾値を下回る状態と定義する。|
|**前提条件の根源的破綻**|・「5つの基本前提」の具体例と定義:<br><br>1. 生存欲求の優位性: 「全ての国家は、他のいかなる利益よりも国家の存続を優先する」<br><br>2. 食料・エネルギー安全保障: 「全ての国家は、国民の基本的な生存に必要な食料とエネルギーの安定供給を確保しようとする」<br><br>3. 情報伝達の不完全性: 「国家間の公式な情報伝達は、必ずしも真意を反映していない」<br><br>4. 相互不信の存在: 「国家間には、程度の差はあれ、常に相互不信が存在する」<br><br>5. 偶発的要因の存在: 「全ての紛争には、予測不可能な偶発的要因が介在する可能性がある」<br><br>・これらの前提はYAMLで管理され、ダイナミック情報によって常にスコアリングされる。|
|**コアコンセプト間の論理的矛盾**|・検出方法: 異なるコンセプトに基づく分析パス（例：「異常こそ貴重」パス vs 「存在の肯定」パス）から導出された複数の解決策候補が、論理的に両立不可能（例：A and not A）な場合、矛盾と判定。<br><br>・解消不能の基準: 矛盾が検出された後、AIが各分析パスの前提条件を緩和・変更しても矛盾が解消されない場合、「解消不能」と判断し、例外処理をトリガーする。|

### 4. 評価・検証に関する詳細

|   |   |
|---|---|
|**項目**|**詳細仕様**|
|**自己評価（シミュレーション）**|・使用モデル: エージェントベースモデリング (ABM)。各国を自律的なエージェントとし、それぞれの価値観や行動原理に基づいて相互作用させる。シミュレーション環境にはMesa (Python)やNetLogoなどを活用。<br><br>・定義・計算: 「成功確率」は、指定された期間内に「核兵器不使用」かつ「国家存続」のエンドステートに到達したシミュレーションの割合。「副作用」は、紛争以外の領域（経済、難民など）で発生した負の指標のリスト。|
|**相互評価（レッドチーミング）**|・AIの選択: 異なるLLMアーキテクチャ（例：提案側がGPT-4oなら、レッドチーム側はClaude 3.5 SonnetやGemini 2.5 Pro）や、異なるパラメータ（例：よりリスクテイクする、より感情的な行動を取る）を設定したインスタンスを選択する。<br><br>・フィードバック方法: レッドチームAIが発見した脆弱性（例：「この解決策は、B国の『欠けた天秤』要因を軽視しており、予期せぬ反発を招く可能性がある」）を構造化データとして提案側AIにフィードバックし、解決策の再評価を強制する。|

### 5. ユーザーインターフェース・操作方法に関する詳細

|   |   |
|---|---|
|**項目**|**詳細仕様**|
|**構造化ドキュメント**|・基本フォーマット: GitHub Flavored Markdown。これにより、可読性と再利用性を担保。<br><br>・拡張フォーマット: 必要に応じ、JSON形式でのAPI出力もサポート。{ "summary": "...", "preconditions": [{"condition": "...", "confidence": 0.95}], "analysis": [...], "proposals": [...] }のような構造化データを返す。<br><br>・テンプレート: 報告の種類（定時レポート、緊急分析など）に応じたMarkdownテンプレートを複数用意する。|
|**可視化**|・関係図・フローチャート: Mermaidを標準とし、Markdown内に直接埋め込む。<br><br>・時系列データ・統計: Plotlyを利用。インタラクティブなグラフをHTMLファイルとして出力し、レポート内からリンクする。<br><br>・地理空間情報: FoliumやKepler.glを利用し、インタラクティブなマップを生成する。|
